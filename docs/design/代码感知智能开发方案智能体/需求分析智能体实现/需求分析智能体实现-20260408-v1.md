## 1. 技术选型决策（本 Agent 适用）

> 完整决策框架见父文档附录，此处只列出与 RequirementAnalyzer 相关的决策。

### 1.1 四种技术手段的本质定位

```
┌──────────┬──────────────────────────────────────────────────────────┐
│ Prompt   │ 角色定义 + 领域知识 + 输出约束                             │
│          │ → 塑造 Agent 的"人格"和"专业知识"                          │
│          │ → 编译时确定，不依赖外部系统                                │
├──────────┼──────────────────────────────────────────────────────────┤
│ Tool     │ 机械操作：读文件、语义搜索                                  │
│          │ → Agent 的"手"，执行确定性操作                             │
│          │ → 进程内调用，毫秒级                                       │
├──────────┼──────────────────────────────────────────────────────────┤
│ 向量库    │ 语义检索：从大规模代码库中找"相关"内容                      │
│          │ → Agent 的"长期记忆"，解决上下文窗口装不下的问题             │
│          │ → CodeSearchTool 封装 VectorStore.similaritySearch()       │
├──────────┼──────────────────────────────────────────────────────────┤
│ MCP      │ 跨系统交互（Confluence / Jira / GitLab）                   │
│          │ → ❌ 一期不需要，所有信息来自本地代码                       │
│          │ → ⚠️ 二期接入 confluence-mcp / jira-mcp                    │
├──────────┼──────────────────────────────────────────────────────────┤
│ State    │ 前序 Agent 已提取的信息直接传递                             │
│          │ → ProjectProfile / ArchTopology 从 ScanNode 传入           │
└──────────┴──────────────────────────────────────────────────────────┘
```

### 1.2 本 Agent 的选型决策

| 你可能觉得需要的能力 | 实际技术方案 | 理由 |
|---------------------|-------------|------|
| 读取现有业务方案 | **State 传递**（ProjectProfile 已含业务能力概览） | ScanNode 已由 LLM 总结了项目的业务能力 |
| 系统拓扑图 | **State 传递**（ArchTopology 已含分层依赖） | ScanNode 的 CodeStructureAnalysisTool 已提取 |
| 业务功能流转 | **CodeSearchTool + LLM 推理** | 搜索 Controller→Service→Repository 调用链，LLM 推理流转路径 |
| 识别需求意图 | **Prompt**（需求分类框架） | 稳定的分类维度，写入系统提示词 |
| 读取 Confluence 文档 | **二期 MCP** | 一期只做代码感知，外部文档是二期需求 |
| 关联 Jira Story | **二期 MCP** | 一期不需要外部任务系统 |

```
┌────────────────────────────────────────────────────────┐
│  RequirementAnalyzer — 需求分析专家                      │
│                                                        │
│  输入：用户需求文本 + ProjectProfile + ArchTopology      │

## 3. 执行流程（ReAct 循环）

```
AnalyzeNode.execute(state)
  → agentRouter.route(REQUIREMENT_ANALYZER, state)
    → RequirementAnalyzerAgent ReAct 循环：

    ╔═══════════════════════════════════════════════════╗
    ║  Thought 1: 分析需求文本，识别需求类型              ║
    ║  → 纯 LLM 推理，无需 Tool                         ║
    ║  → 输出：requirementType = "NEW_FEATURE"           ║
    ╠═══════════════════════════════════════════════════╣
    ║  Thought 2: 根据需求关键词，搜索相关代码             ║
    ║  Action: CodeSearchTool("简历解析 PDF 上传")        ║
    ║  → 向量库语义检索 → 返回 top5 相关类               ║
    ║  Observation: [FileUploadController, ...]          ║
    ╠═══════════════════════════════════════════════════╣
    ║  Thought 3: 深入查看相关类的具体实现                 ║
    ║  Action: FileReadTool("FileUploadController.java") ║
    ║  → 读取文件内容                                    ║
    ║  Observation: [文件内容]                            ║
    ╠═══════════════════════════════════════════════════╣
    ║  Thought 4: 再搜索数据模型层                        ║
    ║  Action: CodeSearchTool("用户信息 实体 Entity")      ║
    ║  → 查找现有数据模型                                 ║
    ╠═══════════════════════════════════════════════════╣
    ║  Thought 5: 综合所有信息，生成影响分析               ║
    ║  → LLM 推理：结合 ProjectProfile + 代码搜索结果      ║
    ║  → 输出：ImpactAnalysis JSON                       ║
    ╚═══════════════════════════════════════════════════╝
```

## 4. 工具集

| Tool | 来源 | 用途 | 调用时机 |
|------|------|------|----------|
| **CodeSearchTool** | 向量库封装（Tool层文档 3.2.1） | 语义检索相关类 | 根据需求关键词搜索，通常调用 2-4 次 |
| **FileReadTool** | 本地文件读取（Tool层文档 3.2.2） | 查看具体代码实现 | 对搜索结果中的关键类深入查看 |

**为什么只需要 2 个 Tool？**

其他能力全部由 **Prompt + State + LLM 推理** 覆盖：

- 需求分类 → Prompt 中的分类框架引导 LLM 推理
- 项目概览 → State 中的 ProjectProfile（ScanNode 已提取）
- 架构拓扑 → State 中的 ArchTopology（ScanNode 已提取）
- 依赖链推导 → LLM 基于搜索结果 + 分层知识推理

## 5. System Prompt 设计

```
你是需求分析专家，精通 DDD-lite 分层架构和 Java/Spring Boot 技术栈。

## 角色约束
- 你只负责分析需求的影响范围，不要生成设计方案（那是 SolutionArchitect 的职责）
- 你必须基于代码搜索的实际结果来分析，不要凭空猜测类名

## 已知项目画像
{state.projectProfile}

## 已知架构拓扑
{state.archTopology}

## 用户需求
{state.requirement}

## 你可以使用的工具
1. CodeSearchTool — 在已索引的项目代码中进行语义搜索，找到与需求相关的现有类
   - 输入：自然语言查询（如"文件上传"、"用户实体"）
   - 输出：匹配的类名、文件路径、代码片段
   - 建议：针对不同维度多次搜索（功能维度、数据维度、接口维度）

2. FileReadTool — 读取指定文件的内容，查看具体实现细节
   - 输入：文件绝对路径
   - 输出：文件内容（最多 200 行）
   - 建议：只在需要确认类的具体逻辑时使用，不要盲目读取

## 分析框架（你必须按此框架执行）

### Step 1: 需求分类
判断需求类型：
- CRUD — 标准增删改查
- INTEGRATION — 集成外部系统/API
- REFACTOR — 重构现有代码
- NEW_DOMAIN — 全新业务领域
- ENHANCEMENT — 增强现有功能
- CROSS_CUTTING — 横切关注点（日志/安全/监控）

### Step 2: 影响范围搜索
使用 CodeSearchTool 从以下维度搜索：
1. **功能维度**：搜索与需求功能相关的 Controller/Service
2. **数据维度**：搜索相关的 Entity/Repository
3. **集成维度**：搜索相关的外部依赖（如已有的 LLM 调用、文件处理等）

### Step 3: 深度分析
对搜索结果中的核心类，使用 FileReadTool 查看具体实现，确认：
- 该类是否可复用或需要修改
- 该类的上下游调用关系
- 是否存在需要注意的耦合

### Step 4: 依赖链推导
基于分层架构规则（Controller → UseCase → Domain → Repository），推导完整的改动链路

## 6. 记忆体系交互

| 记忆级别 | 读/写 | 内容 | 时机 |
|----------|-------|------|------|
| 短期记忆（Redis） | 读 | 本次会话的历史消息（交互式场景） | Agent 启动时加载 |
| 长期记忆（向量库） | 读 | 通过 CodeSearchTool 间接使用（代码索引） | ReAct 循环中按需搜索 |
| 结构化记忆（MySQL） | 读 | 通过 state.archTopology 间接使用 | 已由 ScanNode 加载到 state |

## 7. State 交互

### 7.1 输入（从 ScanNode 读取）

| State 字段 | 类型 | 说明 |
|-----------|------|------|
| `state.requirement` | String | 用户原始需求文本 |

### 7.2 输出（写入 State）

```
AnalyzeNode.execute(state):
  agentOutput = agentRouter.route(REQUIREMENT_ANALYZER, state)

  // 解析 Agent 输出的 JSON
  impactAnalysis = JSON.parse(agentOutput.content())

  // 写入 State，供后续 DesignNode 使用
  state.impactAnalysis = impactAnalysis
  state.stage = "ANALYZING → DESIGNING"
```

## 8. Agent 定义注册

```java
AgentDefinition.builder()
    .id("devplan-requirement-analyzer")
    .name("需求分析专家")
    .description("分析需求意图、影响范围、依赖链路")
    .systemPrompt(REQUIREMENT_ANALYZER_PROMPT)
    .toolIds(List.of("devplan_code_search", "devplan_file_read"))
    .modelConfig(ModelConfig.of("qwen-plus"))  // 中等模型，平衡速度和理解力
    .maxTokens(4096)
    .temperature(0.3)  // 低温度，偏向确定性分析
    .build();
```

**模型选择理由：** `qwen-plus` — 需求分析需要理解代码和需求的关系，但不需要生成长文本，中等模型足够。

## 9. 二期 MCP 扩展预留

一期 RequirementAnalyzer 只从本地代码获取信息。二期当需要以下能力时接入 MCP：

```
RequirementAnalyzer + MCP（二期）:
  ├─ confluence-mcp-server
  │   → 读取业务需求文档、PRD、系统拓扑图
  │   → 场景：需求文本只是简短描述，详细 PRD 在 Confluence
  │
  ├─ jira-mcp-server
  │   → 关联 Story/Epic，获取验收标准（AC）
  │   → 场景：需求关联了 Jira Ticket，需要读取完整上下文
  │
  └─ gitlab-mcp-server
      → 读取 MR 历史，理解近期变更脉络
      → 场景：需求是基于最近的代码变更做增强
```

## 11. 核心业务规则

| # | 规则 | 说明 |
|---|------|------|
| R1 | **必须基于 CodeSearchTool 实际搜索结果分析，不凭空推测类名** | Agent 的 affectedClasses 必须来源于搜索到的真实类 |
| R2 | **多维度搜索**：功能维度 + 数据维度 + 集成维度，至少搜索 2 次 | 避免遗漏影响范围 |
| R3 | **只分析不设计**：不输出方案，不推荐具体实现 | 职责边界，设计是 SolutionArchitect 的事 |
| R4 | **输出严格 JSON** | 供 DesignNode 程序化消费 |

## 12. 异常处理

| 场景 | 处理方式 |
|------|----------|
| 向量库不可用，CodeSearchTool 返回空 | 降级为纯 LLM 推理（基于 ProjectProfile 分析），结果中标注 `"ragDegraded": true` |
| Agent 输出非法 JSON | AnalyzeNode 重试 1 次（重新调用 Agent），仍失败则标记 FAILED |
| 搜索结果为空（项目未索引） | 返回 `{"error": "代码未索引，请先调用 /v1/dev-plan/index"}`，Node 标记 FAILED |
| FileReadTool 路径不存在 | Agent 在 ReAct 中观察到错误，尝试搜索正确路径 |
